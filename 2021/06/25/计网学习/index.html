整理了一些面试常问到的计网问题

Http协议

网络都是建立在TCP/IP协议族之上的，属于7层模型应用层，其他常见包括http、ftp、ssh、dns等

三次握手

  • 第一次，客户端发送一些数据到 server 端，要求建立连接
  • 第二次，server 端收到，然后发送一些数据回去，表明可以建立连接
  • 第三次，客户端收到并检查返回的信息，若信息正确，再发送数据到 server 端，告知建立连接成功

http1.1版本支持持久链接keep-alive， 只要双方不明确终止链接，链接就会保持，每次发送数据不再需要三次握手

请求报文内容

  • Method
  • Url
  • 协议版本
  • Head 头部字段
  • 内容实体

Method

  • GET：获取
  • POST：新增
  • PATCH/PUT：提交
  • HEAD：获取 Head 信息，不返回报文主题
  • DELETE：删除
  • OPTIONS、TRACE、CONNECT

响应报文内容

  • 协议版本
  • 状态码
  • 状态码原因短语
  • Head 头部字段
  • 响应内容

状态码

  • 1xx 请求正在处理
  • 2xx 成功
  • 3xx 重定向
  • 4xx 客户端错误
  • 5xx 服务端错误

2xx

  • 200成功
  • 204成功、但没有返回实体
  • 206范围请求

3xx

  • 301永久重定向
  • 302临时重定向
  • 304协商缓存，资源找到，但本地缓存未过期，不会返回任何主体

4xx

  • 400请求语法错误，服务器无法理解
  • 401请求需认证，登录
  • 403没有访问权限，服务端理解请求，但拒绝执行此请求，如防盗链
  • 404资源未找到
  • 405客户端请求方法被禁止

5xx

  • 500服务端在执行时发生错误
  • 501服务端不支持请求的功能，无法完成请求
  • 502作为网关或者代理时，从远程服务器收到了一个无效的响应
  • 503服务端超负载
  • 504作为网关或者代理时，为及时从远端获取请求

Head 头部字段

通用字段

  • Cache-Control:max-age=3600
  • Connection: keep-alive
  • Date 获得服务端时间

请求头字段

  • Accept客户端可接受媒体类型
  • Accept-Encoding客户端可接受的压缩类型
  • Accept-Language
  • Host域名及端口号，不包含协议
  • If-Match协商缓存标识
  • If-Modified-Since协商缓存标识
  • Referer请求来源于哪里
  • User-Agent客户端信息，没有跨域限制
  • Cookie存在跨域限制，只有同源才会传递

响应头字段

  • Age缓存持续时长，Age:300表明该资源是300s之前创建与Cache-Control:max-age 比对
  • ETag服务器为资源分配的唯一标识，资源更新也会随之更新，用于协商缓存，对应If-Match
  • Location重定向地址，当状态码为3xx时
  • Server服务器信息
  • Set-Cookie本次请求中服务端设置的cookie信息

TCP 三次握手流程

  1. 客户端发送 syn 包到服务器，进入 syn_send 状态，syn 意为同步序列号
  2. 服务器收到 syn 包进行确认，同时也发送一个 SYN 包（SYN + ACK）进入 SYN_RECV 状态
  3. 客户端接收到服务器的 SYN + ACK 包，向服务器发送确认包 ACK ，发送完毕后，客户端今儿服务器进入 连接成功状态，完成三次握手

为何要三次握手

在第二次握手后，若客户端没有收到回应，客户端认为链接没有建立，而服务端在发送完后会认为链接已经成立，并保存必要资源，入遇大量此类情况，会导致服务器崩溃。

TCP 四次挥手

由于 TCP 为全双工，因此每个方向都需要单独进行关闭，当一方完成数据发送任务后可发送一个 FIN 来终止这个方向上的连接，另一方接收到 FIN 信号知晓这一方向不再有数据流，但自身仍能发送数据，首先进行关闭的一方执行主动关闭，另一方则是被动关闭

  1. 客户端发送一个 FIN，终止客户端到服务器的数据发送
  2. 服务端收到该 FIN，返回一个 ACK，确认序号为收到序号加1，一个 FIN 占用一个序号
  3. 服务端关闭与客户端连接，发送一个 FIN 给客户端
  4. 客户端返回 ACK 确认，将确认序号设置为收到序号加1

为何四次挥手

因为服务端的LISTEN状态下的SOCKET当收到SYN报文的建连请求后，它可以把ACK和SYN (ACK起应答作用，而SYN起同步作用)放在一个报文里来发送。但关闭连接时，当收到对方的FIN报文通知时，它仅仅表示对方没有数据发送给你了；但未必你所有的数据都全部发送给对方了，所以你可以未必会马上会关闭SOCKET,也即你可能还需要发送一些数据给对方之后，再发送FIN报文给对方来表示你同意现在可以关闭连接了，所以它这里的ACK报文和FIN报文多数情况下都是分开发送的。

打开一个网址后发生了什么

  1. 检查浏览器缓存、检查本地 hosts 文件是否存在这个网址映射，如果有，调用这个 IP 地址映射，解析完成
  2. 如果没有，则查找本地 DNS 解析器缓存是否有这个网址映射，如果有，返回映射，解析完成
  3. 如果没有，则查找填写或分配的的首选本地 DNS 服务器，检查要该域名是否包含在本地配置区域资源中：是，返回解析结果，查询结束，此解析具有权威性；否，但服务器缓存了此网址映射关系，返回解析结果，查询结束，此解析不具有权威性
  4. 如果本地 DNS 服务器失效，则采取两种方式：
  • 转发模式-迭代：本地 DNS 将请求发送到13台根 DNS 服务器，判断由谁管理并返回一个负责该顶级域名服务器的 IP ，本地 DNS 服务器收到该 IP 后向其发送请求，该服务器如果无法解析，则继续找负责这个域名的下一级 DNS 服务器，直到找到映射，见解析结果返回本地 DNS 服务器，由本地 DNS 服务器返回解析结果，解析完成
  • 转发模式-递归：DNS 服务器将请求妆发至上一级 DNS 服务器，如果上一级 DNS 服务器不能解析，则继续向上请求。最终得到解析结果
  1. 得到目标服务器 IP 及端口号后，调用系统库函数 socket 请求一个 TCP 流套接字，客户端向服务器发送 HTTP 请求报文
  2. 服务端解析请求报文，发送响应报文
  3. 关闭连接，TCP 四次挥手
  4. 客户端解析响应，渲染 DOM，CSSOM，合并树，显示 HTML

TCP/IP 五层模型

  1. 应用层：客户端发送HTTP请求报文。
  2. 传输层：(加入源端口、目的端口)建立连接。实际发送数据之前，三次握手客户端和服务器建立起一个TCP连接。
  3. 网络层：(加入IP头)路由寻址。
  4. 数据链路层：(加入frame头)传输数据。
  5. 物理层：物理传输bit。

OSI 七层模型

  1. 应用层
  2. 表示层
  3. 会话层
  4. 传输层
  5. 网络层
  6. 数据链路层
  7. 物理层

七层/五层模型应用

  1. 应用层：TCP【HTTPS（万维网）、HTTP、TELNET（远程终端接入）、FTP（文件传输）、SMTP/POP3（电子邮件）】、UDP【TFTP（文件传输）、DNS（域名转换）】、
  2. 表示层：LPP、NBSSP
  3. 会话层：SSL、TLS、DAP、LDAP
  4. 传输层：TCP、UDP
  5. 网络层：IP、ICMP、RIP、IGMP、OSPF
  6. 数据链路层：以太网、网卡，交换机、令牌环、PPP、PPTP、L2TP、ARP、ATMP
  7. 物理层：物理线路，光线，无线电

GET、POST 区别

  • GET 只能进行 URL 编码；POST 多种编码方式
  • GET 存在缓存；POST 无
  • GET 通过 URL 传递数据，但长度存在限制

TCP、UDP 区别

  • TCP 面向连接；UDP 无连接
  • TCP 提供可靠服务，保证无差错，不丢失，有序到达；UDP 尽最大努力传输，不保证可靠性
  • TCP 面向字节流，把数据当作一连串无结构字节流；UDP 面向报文，应用层发送多长的报文 UDP 就照样发送，没有拥塞控制，及时网络出现阻塞，也不会使原主机发送速率降低
  • TCP 仅支持点到点；UDP 支持一对一，一对多，多对一，多对多
  • TCP 首部开销20字节，UDP 仅8字节
  • TCP 的逻辑通信信道是全双工可靠信道，UDP 则是不可靠信道

TCP 为什么可靠

  • 确认重传机制
  • 数据排序
  • 流量控制
  • 拥塞控制：慢启动，拥塞避免，快速重传，快速恢复

IP 地址分类

  • A类：0.0.0.0-126.255.255.255，默认网络掩码为：255.0.0.0，A类地址分配给规模特别大的网络使用，
  • B类：128.0.0.0-191.255.255.255，默认网络掩码为歎：255.255.0.0，B类地址分配给一般的中型网络
  • C类：192.0.0.0-223.255.255.255，默认网络掩码是：255.255.255.0，C类地址分配给小型网络，如局域网
  • D类：广播地址，共特殊协议向选定的节点发送信息使用。
